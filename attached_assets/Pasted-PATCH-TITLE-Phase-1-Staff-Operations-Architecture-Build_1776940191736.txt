PATCH TITLE
Phase 1 — Staff Operations Architecture Build

OBJECTIVE
Build the backend architecture and route/navigation scaffolding for a configurable Staff Operations module inside the Operations section.

This module must support:
- staff management
- roster/shift setup
- work area allocation
- break rules
- cleaning task allocation
- attendance logging
- sickness/replacement handling
- audit/change history

This is NOT Smash Brothers Burgers specific.
Do NOT hardcode SBB shift times, roles, stations, break rules, or cleaning tasks.
Build this as a configurable multi-business / multi-location operations system from day one.

PHASE RULE
This patch is Phase 1 only:
1. Architectural Build
Do NOT build the final polished UI yet.
Do include:
- schema
- migrations
- backend routes
- services
- validation
- sidebar navigation entry
- route registration
- placeholder pages that render successfully

Do NOT do Phase 2 UI polish or Phase 3 test suite in this patch.

NON-NEGOTIABLE RULES
1. Additive only
Do not break or refactor unrelated systems.
Do not disturb:
- Daily Sales / Stock flows
- existing Operations features
- Bob / Dashboard functionality
- existing protected forms or reporting flows

2. Multi-business / multi-location
All operational records must be scoped to the existing business/location/tenant pattern already used in the app.
If a canonical business/location model already exists, use it.
Do NOT create duplicate tenancy logic if one already exists.

3. No assumed shift times
Shift times must be entered/configured in the app.
Do NOT assume:
- 5:45 PM to 2:15 AM
- 5:00 PM prep
- 3 staff max
- 45 minute break + 2 x 15 minute breaks
These may be seeded as examples later, but runtime logic must come from settings/config tables.

4. Multiple shifts per day
The system must support more than one shift on the same date.
Do NOT design this as one roster row per date only.

5. Service-layer logic
Do not bury all logic inside route handlers.
Use route module + service layer + validation.

6. Audit trail required
Any meaningful roster change must be loggable in a shift change log table/service.
