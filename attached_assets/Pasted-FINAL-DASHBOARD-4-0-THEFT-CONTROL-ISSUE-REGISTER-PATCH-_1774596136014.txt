==================================================
AUTO-CREATION RULES
==================================================

Use the existing working data already shown in Sales & Shift Analysis / stock reconciliation.

Auto-create issues for these cases:

1. Missing usage build
Condition:
- analysis/stock-usage for shift date returns not_built = true
Create:
- issue_type: MISSING_USAGE_BUILD
- category: SYSTEM
- severity: CRITICAL

2. Buns variance beyond threshold
Thresholds:
- warn > 5
- critical > 10
Create:
- issue_type: BUNS_VARIANCE
- category: STOCK
- severity according to threshold

3. Meat variance beyond threshold
Thresholds:
- warn > 500g
- critical > 1000g
Create:
- issue_type: MEAT_VARIANCE
- category: STOCK

4. Drink variance by type beyond threshold
Thresholds:
- warn > 2
- critical > 4
Create one issue per drink type when breached
- issue_type: DRINK_VARIANCE
- category: STOCK

5. Invalid stock inputs
Examples:
- closing > opening + received
- physical used < 0
- impossible totals
Create:
- issue_type: INVALID_STOCK_INPUT
- category: DATA
- severity: HIGH or CRITICAL depending on impact

6. Missing fries data
If shift analysis cannot validate fries because no staff-side actual exists:
- issue_type: MISSING_FRIES_DATA
- category: DATA
- severity: MEDIUM

7. Meat split ambiguity
If actual form only has combined meatWeightG and no beef/chicken split for a shift where chicken/beef are both used:
- issue_type: MEAT_SPLIT_AMBIGUITY
- category: DATA
- severity: MEDIUM

8. POS vs Form mismatch
If Total Sales mismatch exists and exceeds practical tolerance:
- issue_type: POS_FORM_MISMATCH
- category: FINANCIAL
- severity based on delta

IMPORTANT VALIDATION RULES
==================================================

1. No duplicate issue spam
If an issue already exists for same:
- shift_date
- issue_type
- source_ref or source_page
- same key metadata
then update timestamp/metadata if needed, do not create endless duplicates.

2. Status behavior
- OPEN = newly detected
- IN_PROGRESS = someone is working it
- RESOLVED = fix/action applied but not verified
- CLOSED = verified final

3. Closed issues must remain visible in the register
They are part of history.

4. When a previously detected issue no longer applies, do NOT auto-delete it.
Optional safe behavior:
- leave OPEN until human resolves/closes
or
- mark system note in metadata that the live condition is no longer present
But do not silently remove issue history.
